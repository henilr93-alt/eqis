📖 SearchPulse Engine Rule Book

Defines how search duration is calculated, rating thresholds, escalation rules, and failure classifications.

⏱ Search Duration Measurement Rules

Principle: Duration = time from Search button click until loading on the results page stops. This is the exact wait time an agent experiences before seeing usable results.
StepAll 4 Engines (Flight DOM, Flight INTL, Hotel DOM, Hotel INTL)
▶ Timer STARTSImmediately after "Search Flight" or "Search Hotel" button is clicked.
searchSubmitTime = Date.now() — right after clickSearchFlight() / clickSearchHotels()
⏳ Phase 1: NavigationWait for URL to change to results page:
• Flights: /flights/oneway or /flights/roundtrip (10s timeout)
• Hotels: /hotels/search-results (15s timeout)
If URL doesn't change → FAILED (form didn't submit)
⏳ Phase 2: Results LoadingWait for result cards or "Showing (N)" text to appear:
• Flights: .accordion_container cards or "Showing (N)" text (45s timeout)
• Hotels: hotel cards or "Showing (N) Hotels" text (60s timeout)
Also detects "No results found" for zero-result searches
⏳ Phase 3: API Requests CompleteActively tracks all XHR/Fetch/Document HTTP requests in real-time. Waits until 0 active API requests for 2 consecutive seconds.
This ensures ALL flight/hotel data has been fetched from Etrav's API before stopping. The loading UI on screen disappears when API calls finish — this detects exactly that.
• Flights: 45s timeout  • Hotels: 60s timeout
⏹ Timer STOPS1.5 seconds after all API requests complete (2s quiet period).
The 1.5s buffer catches any delayed final DOM renders after all data has loaded.
result.loadTimeMs = Date.now() - searchSubmitTime
✅ INCLUDED in durationSearch button click → URL change → API data loading → result cards rendering → network idle (all API calls complete) + 1.5s buffer
❌ NOT includedPage navigation to /flights or /hotels, login, autosuggest form filling, date picker interaction, cabin class selection, pax selection — these happen BEFORE the search button click
Example: If the search button is clicked at video timestamp 14s and loading stops at 28s → Search Duration = 14 seconds. The video duration before 14s (login, form filling) is NOT counted.

✈️ Round Trip Fare Ticker — Flight Engines

Flight DOM & Flight INTL: When searching Round Trip, there is an additional ticker on the right side of the form labeled "RoundTrip Fare" with a check mark, which appears only for round-trip searches. By default, this ticker is checked (enabled).
AspectBehavior
When visibleOnly when "Round Trip" trip type is selected (not for One Way or Multi City)
Default stateChecked (enabled) by default — returns combined round-trip fare bundles
Toggle patternEvery alternate round-trip search must uncheck this ticker before submitting. Pattern: 1st RT search = checked, 2nd RT search = unchecked, 3rd RT search = checked, 4th RT search = unchecked, and so on.
Why alternateTests both "RoundTrip Fare" mode (combined fare) and "Special Return" mode (separate one-way fares displayed together) — different result sets, different rendering paths, different API endpoints
URL impactChecked: search_type=SPECIAL_RETURN with is_special_return=true  |  Unchecked: search_type=ROUNDTRIP with is_special_return=false
TrackingCounter persisted in state/pulseHistory.json as roundTripCount. Engine increments on every round-trip search and toggles checkbox based on parity (even = checked, odd = unchecked).
One Way / Multi CityNot applicable — ticker is hidden for non-round-trip searches
Test coverage rationale: Both modes use different Etrav backend code paths. Testing only one mode would miss bugs in the other. Alternating ensures equal sample size for both fare display modes across the 24h window.

🔎 Autosuggest Behavior — Hotel Engines

Hotel DOM & Hotel INTL: When filling the destination field, the system must wait for Etrav's autosuggest API to display the dropdown list, then select the matching destination by clicking on it from the dropdown. Direct typing without clicking a suggestion will not set Etrav's locationId, and the search will fail to submit.
StepBehavior
1. Click inputClick the "Hotel name or Destination" input to focus it
2. Clear fieldUse input.fill('') for clean React state reset
3. Wait for clearWait until previous suggestions disappear from DOM (max 3s)
4. Type destinationType the destination text (e.g., "Krabi", "Tokyo", "Dubai") with 150ms delay per character
5. Wait for APICritical: Wait 1500ms for Etrav's debounced autosuggest API to fire and return the suggestions list
6. Wait for dropdownWait for .react-autosuggest__suggestion elements to appear (max 8s) + 500ms for full list population
7. Click matchFind the suggestion matching the target destination and click it from the dropdown. If no match, click the first suggestion as fallback.
8. Verify selectionRead back the input value and confirm it contains the target destination. If mismatch, retry up to 3 times.
9. Bail-out on failureIf all 3 attempts fail → mark as AUTOSUGGEST_DOWN (Etrav platform issue) and skip the rest of form filling — do NOT click Search
Why clicking is required: Etrav's hotel search relies on a locationId in the URL (e.g., locationId=213394 for Dubai). This ID is set only when a suggestion from the dropdown is clicked — typing the destination text alone does not set it. Searching without a valid locationId returns the form page (no results URL), which is recorded as SPF.

SLIDE 01 · THE PLATFORM
What is EQIS?
EQIS (ETRAV QA Intelligence System) is an always-on testing platform that behaves like a team of virtual agents continuously booking flights and hotels on new.etrav.in. Every run is captured, every click is measured, every API call is costed, and every bug is surfaced the moment it appears.
WHY IT EXISTS
Manual QA can't match the scale of a live B2B travel platform. EQIS runs the journeys a real travel agent would run — every 10 minutes, 30 minutes, and hourly — and flags everything that breaks before customers see it.
WHAT IT MEASURES
Load times, filter accuracy, API error rates, zero-result routes, booking-flow completion, payment page health, Zipy session anomalies, and per-scenario cost.

// Every engine runs through guardedRun() which checks:
// 1. Is the engine paused? (engineState.json)
// 2. Is FRAKA awake? (frakaState.json)
// Only if both pass does the runner fire.

function guardedRun(engineName) {
  return () => {
    const state = loadEngineState();
    if (state[engineName] === 'paused') return;
    if (runners[engineName]) runners[engineName]();
  };
}

SECTION 06
Autonomous Builder — coder.js
357 lines · Plan → Write → QA → Rollback → Report pipeline
// runCoderBuild(task, { inlineFiles }) — full autonomous cycle

Step 1: PLAN    → Sonnet receives task + project tree + CEO directives
                 → Returns JSON: { plan, files[], followups[], ceoNote, techNote }

Step 2: WRITE   → Each file goes through codeReader.writeProjectFile()
                 → Auto-backup to state/fraka/code-backups/TIMESTAMP__path.bak
                 → Max 8 files per cycle, 500KB per file

Step 3: QA      → codeQA.runFileQA() on every written file:
                 → JS: syntax parse + require() load + brace balance
                 → JSON: JSON.parse validation
                 → HTML: tag balance + inline <script> check
                 → ALL: non-empty + forbidden credential patterns

Step 4: ROLLBACK → If ANY file fails QA:
                 → Restore all files from backups
                 → Delete any newly-created files
                 → Status: 'failed-qa' or 'rolled-back'

Step 5: REPORT  → Post to Tech chat: full file list, QA results, rollback log
                 → Post to CEO chat: 1-sentence executive summary
                 → Persist to state/fraka/buildHistory.json

Path Guards — codeReader.js

// BLOCKLIST — FRAKA can NEVER read or write these:
const BLOCKLIST_PATTERNS = [
  /^\.env/,                          // credentials
  /^\.git(\/|$)/,                     // version control
  /^node_modules(\/|$)/,              // dependencies
  /^state\/fraka\/chats(\/|$)/,       // chat history
  /^state\/fraka\/proposals\.json$/,   // proposal store
  /^state\/fraka\/developments\.json$/,// dev backlog
  /^logs(\/|$)/,                      // log files
  /^reports(\/|$)/,                   // generated reports
  /secret/i, /password/i, /credential/i, // credential patterns
];

// ALLOWED extensions only:
.js .mjs .cjs .json .md .txt .html .css .svg .yml .yaml .sh .ts

// Every write creates a backup FIRST:
function writeProjectFile(relPath, content, operation) {
  guard(relPath);                      // blocklist + extension check
  if (existed) backupFile(relPath);    // → code-backups/TIMESTAMP__path.bak
  fs.writeFileSync(absPath, content);  // atomic write
}
